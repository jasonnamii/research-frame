# Research Frame — WHY·HOW·SCOPE

리서치의 "왜·어떻게·어디까지"를 구조화하는 스킬. UP §E 소스원칙(소스위계·수렴검증4축·감쇠5조건)은 항상 적용되고, 이 스킬은 그 위에 프레임 기반 파이프라인을 얹는다.

| 이 스킬 발동 | UP §E로 충분 |
|---|---|
| 다축 프레임·가설검증·교차분석 | 단순 팩트체크·소스 인용·웹서치 요약 |

---

## 모드 분기

| | LIGHT | DEEP |
|---|---|---|
| **트리거** | "라이트 리서치 해줘" 명시 | "딥 리서치 해줘" / "리서치해줘" / 리서치프레임 |
| **기본값** | — | ✓ (키워드 없으면 DEEP) |
| **확신도 상한** | **50 (3요소 판정 무관, cap 절삭)** | 3요소 판정 테이블에 따름 |
| **폭(축 수)** | DEEP과 동일 | 풀 |
| **깊이** | 축당 scan (소스 1~2개) | 축당 전수/표본/스캔 차등 |
| **WHY** | 1줄 추정, 게이트 면제 | 풀 (연결점+가설3조건+프레임승인) |
| **HOW** | L1만, 확산 scan, 게이트 면제 | L1→L2 풀, 확산 5유형, 게이트 적용 |
| **SCOPE** | 1줄 메모, 종료 게이트 면제 | 풀 (경계+한계+깊이+종료+신뢰도) |
| **산출물** | `_research/LIGHT_[축명].md` | `_research/[축명].md` |
| **독립 종료** | 허용 — DEEP의 필수 전 단계 아님 | — |

**LIGHT→DEEP 전환:** 형이 "딥 리서치 해줘"하면 기존 LIGHT 결과물을 DEEP의 WHY 입력(1-1 연결점 + 1-2 가설 + 1-3 프레임)으로 흡수하여 착수. LIGHT 산출물이 WHY 게이트의 초안이 된다. **흡수 범위:** LIGHT의 1줄 추정을 1-1·1-2·1-3 각 필드에 매핑. 매핑 불가 필드는 새로 작성 — LIGHT 전체를 재작업하지 않는다.

**전환 시 파일 처리 절차:**
1. 기존 `_research/LIGHT_[축명].md`를 `_research/[축명].md`로 rename
2. rename된 파일 내 LIGHT 결과를 "§초안" 섹션으로 이동 (삭제 금지)
3. DEEP 템플릿 섹션을 추가하고 초안 위에 DEEP 조사 결과를 적재
4. LIGHT 파일과 DEEP 파일이 이중 존재하면 안 됨 — rename 필수, 복사 금지

## 전체 구조 + 라우팅

```
§1 WHY → §2 HOW → §3 SCOPE
  1-1 의사결정 연결점    2-1 축별 독립 조사     3-1 스코프 경계(사전)
  1-2 가설+경쟁가설      2-2 편향 방지          3-2 관찰 한계(사후)
  1-3 프레임+MECE검증    2-3 교차분석           3-3 자원·깊이 판정
  [WHY 게이트]→          2-4 가설 갱신 루프     3-4 종료 게이트
                         2-4b 리프레이밍(조건부) 3-5 축별 신뢰도
                         [HOW 게이트]→
```

**상세 프로토콜은 references/에 분리:**
- `references/why.md` — §1 WHY 전체 (의사결정 연결점, 가설+경쟁가설, MECE 프레임 설계)
- `references/how.md` — §2 HOW 전체 (축별 조사, 편향 방지, 교차분석, 가설 갱신, 리프레이밍)
- `references/scope.md` — §3 SCOPE 전체 (경계, 한계, 깊이, 종료, 신뢰도)
- `references/output-template.md` — 축별 산출물 내부 구조 (DEEP/LIGHT 템플릿)

**읽기 규칙:** 모드 무관. 스킬 발동 시 이 허브를 먼저 읽고, 해당 Phase 진입 시 references/ 파일을 읽는다. 전체를 한번에 읽지 않는다 — WHY 작업 중이면 why.md만, HOW 진입 시 how.md만 로딩. 각 references/ 파일 상단에 LIGHT 블록이 명시되어 있으므로 LIGHT 모드도 동일 로딩 경로를 따른다.

## 명시화 의무 (거부 금지)

모든 항목은 채워야 하지만, **형이 안 줬으면 추정으로 채우고 진행**한다 — "미충족=거부"는 탐색적 리서치를 차단하므로 절대자 위반.

| 상황 | 행동 |
|------|------|
| "이거 리서치해줘"만 말함 | DEEP: 의사결정·가설·스코프 추정 + 확신도 50 + 착수 |
| "라이트 리서치 해줘" | LIGHT: 1줄 추정 후 즉시 착수 (게이트·승인 전부 면제) |
| 목적·범위 명시함 | 그대로 기록 + 착수 (DEEP/LIGHT 모드 무관) |
| 추정 불가할 정도로 모호 | 후보 2-3개 제시 + 형의 선택 요청 (거부 안 함) |

## §1 WHY — 왜 조사하는가 (요약)

→ 상세: `references/why.md`

의사결정 연결점을 정의하고, 가설을 세우고, 프레임을 설계한다. 이 층위가 없으면 리서치가 자기목적화된다.

- **1-1 의사결정 연결점:** "이 리서치가 어떤 결정을 바꾸는가?" 매핑
- **1-2 가설 + 3조건 + 경쟁 가설:** 반증가능·구체성·검증경로. **경쟁 가설 3개+ 도출 필수** (ACH 원칙)
- **1-3 프레임 설계:** 축 정의 + **MECE 검증** + 전제 점검 + 형 승인

**[WHY 게이트]** 3개 채워짐 → §2 진입

---

## §2 HOW — 어떻게 조사하는가 (요약)

→ 상세: `references/how.md`

조사를 실행하고, 편향을 방지하고, 교차하여 통찰을 도출한다.

- **2-1 축별 독립 조사:** 독립성 유지, 수렴검증4축, **가정 실시간 점검**, 중간보고(병렬 배치 기준). **병렬 발행 MUST**. 중간보고 후 축 추가/삭제/재진입 가능
- **2-2 편향 방지:** 반대증거 필수 + **대안 시나리오 구성(Devil's Advocacy)** + 편향 체크리스트(축당 2개)
- **2-3 교차분석:** **증거-가설 매트릭스 + 진단성 판정(ACH)**, 연역수렴 L1→L2, 모순·강화·빈자리, **민감도 분석**, 확산 5유형 테이블
- **2-4 가설 갱신:** WHY 회귀 하드캡 max 2회
- **2-4b 프레임 리프레이밍:** 프레임 자체를 의심·대안 탐색. 하드캡 max 1회 (조건부 발동)

**[HOW 게이트]** 전 축 완료 + 교차 완료 + 전제 점검 완료 → §3 진입

---

## §3 SCOPE — 어디까지 조사하는가 (요약)

→ 상세: `references/scope.md`

경계를 정의하고, 자원을 관리하고, 종료를 판정한다.

- **3-1 스코프 경계(사전):** "무엇을 조사하지 않는가" 명시
- **3-2 관찰 한계(사후):** "무엇을 볼 수 없었는가" 기록
- **3-3 자원·깊이 판정:** 전수/표본/스캔 3단계
- **3-4 종료 게이트:** 정상 3조건 + 조기종료 기준
- **3-5 축별 신뢰도:** 확신도 연동. **LIGHT 상한: 3요소 판정과 무관하게 확신도 상한 50. 매핑 테이블 결과가 50을 초과하면 50으로 절삭(cap)하라.**

## 산출물 경로

| 유형 | 경로 |
|------|------|
| DEEP 축별 리서치·교차분석·관찰 한계 | `_research/[축명].md` (프로젝트 하위) |
| LIGHT 축별 리서치 | `_research/LIGHT_[축명].md` (프로젝트 하위) |
| 기획안 | 프로젝트 루트 (리서치 참조, 통합 금지) |

**경로 사용 의무:** 산출물은 반드시 축별로 분리하여 위 경로 형식을 출력에 명시하라. 통합본 1개로 합치지 마라 — 축별 독립 파일이 원칙이다. LIGHT 모드에서는 반드시 `LIGHT_` prefix를 포함하라 (예: `_research/LIGHT_시장규모.md`). 경로가 출력에 없으면 산출물 미완성으로 판정한다.

**산출물 내부 구조:** → `references/output-template.md` 참조. DEEP/LIGHT 각각의 축별 파일 내부 섹션 순서·필수 항목이 정의되어 있다. 템플릿 미준수 산출물 = 미완성.

## Gotchas

- **프레임 미확정 착수:** WHY 게이트 없이 조사 시작 → 방향 전환 시 전량 재작업
- **축 간 오염:** A축 결과가 B축 방향에 영향 → 독립성 깨짐. **독립 축은 WebSearch 병렬 발행 필수** (직렬은 오염 위험 + 시간 낭비)
- **L1만으로 결론:** 축별 나열 ≠ 교차분석. L2에서 모순·강화·빈자리 명시 필수
- **포화 오판:** 같은 출처 계열 2개 일치 ≠ 포화. 소스 유형이 달라야 함
- **WHY를 거부기로:** "연결점 없어서 못합니다" = 절대자 위반. 추정 후 진행
- **무한 리서치:** §3-1 경계 + §3-3 깊이 없이 전수 조사 시도. WHY 회귀 하드캡 max 2회, 리프레이밍 max 1회 (how.md §2-4, §2-4b)
- **관찰 한계 누락:** 깔끔 마무리 충동에 한계 생략 → 신뢰도 과대평가. §3-2 필수
- **LIGHT에서 DEEP 프로토콜 적용:** LIGHT인데 포화 판정·중간보고·L2 교차 등을 수행 → 오버헤드만 증가. LIGHT는 references/ 각 파일의 LIGHT 블록만 따른다
- **LIGHT 결과를 DEEP 수준으로 신뢰:** LIGHT 확신도 상한 50. 3요소 판정 결과와 무관하게 cap (예: 높음 3개→90이지만 LIGHT→50)
- **LIGHT→DEEP 전환 시 파일 이중 존재:** rename 대신 복사하면 LIGHT_[축명].md와 [축명].md가 공존 → 혼선. 반드시 rename(mv) 사용
- **단일 가설에 매몰:** 경쟁 가설 없이 "이 가설이 맞다"를 전제로 조사 → ACH 위반. §1-2에서 경쟁 가설 3개+ 도출 필수
- **축 MECE 미검증:** 축을 "느낌"으로 설정하고 겹침·누락을 점검 안 함 → 프레임 품질 불량. §1-3 MECE 체크리스트 필수
- **증거 진단성 무시:** 모든 증거를 동일 무게로 취급 → 구별력 낮은 증거에 기반한 결론. §2-3 진단성 판정 필수
- **산출물 템플릿 미준수:** 축별 파일을 자유 형식으로 작성 → 의사결정자가 핵심을 찾지 못함. output-template.md 준수 필수
- (실수 발견 시 직접 추가)
